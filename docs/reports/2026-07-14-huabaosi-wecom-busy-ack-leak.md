# Huabaosi WeCom Busy Ack Leak

Date: 2026-07-14

Status: read-only production diagnosis; no server files changed

## Current State

A production user saw 阿亮画报师 send:

- `Interrupting current task. I'll respond to your message shortly.`
- `(Response formatting failed, plain text:)`

The screenshot was from the Huabaosi WeCom Bot path, not from the Rust sidecar image
generation worker or the reviewed QiWe image-send adapter.

## Evidence

Read-only SSH checks found the active service:

```text
hermes-gateway-huabaosi.service
ExecStart=/home/ubuntu/.hermes/hermes-agent/venv/bin/python -m hermes_cli.main --profile huabaosi gateway run --replace
WorkingDirectory=/home/ubuntu/.hermes/profiles/huabaosi
```

The fixed interruption text is generated by live Hermes gateway code in
`/home/ubuntu/.hermes/hermes-agent/gateway/run.py` when a message arrives while the same
session is already busy and `busy_input_mode=interrupt`.

The fallback prefix is generated by
`/home/ubuntu/.hermes/hermes-agent/gateway/platforms/base.py` when platform-specific
send formatting fails and Hermes retries as plain text.

The Huabaosi profile has `busy_input_mode: interrupt`. The WeCom adapter has an internal
process filter, but its pattern list covers approval prompts, working traces, tools,
tracebacks, paths, and record-shaped ids; it does not cover the busy-ack interruption
text or the formatting-fallback wrapper.

At the screenshot time window, sanitized journal evidence showed:

```text
13:00:57 [Wecom] Send failed: Timeout sending message to WeCom - trying plain-text fallback
13:01:12 [Wecom] Fallback send also failed: Timeout sending message to WeCom
13:02:48 [Wecom] Send failed: Timeout sending message to WeCom - trying plain-text fallback
13:03:03 [Wecom] Fallback send also failed: Timeout sending message to WeCom
13:04:48 API call failed against the configured custom model provider: Request timed out
```

Public exact-string web search did not identify this as a common third-party API error
message. Local monorepo search did not find the phrase in product code. It is present in
the live Hermes runtime source and tests.

## Root Cause

The user-facing interruption came from Hermes gateway busy-session handling:

1. A previous Huabaosi run was still active or stuck sending.
2. The user sent another message in the same WeCom session.
3. `busy_input_mode=interrupt` generated the busy ack.
4. WeCom send timed out or formatting failed, so `base.py` retried with the
   `(Response formatting failed, plain text:)` prefix.
5. The WeCom internal-process filter did not classify this busy/fallback text as
   internal status, so the message reached the user.

The upstream model-provider timeout likely contributed to the session staying busy, but
the literal text is not returned by that provider. The literal text is from Hermes.

## Risk

This can leak internal runtime state into customer-facing WeCom chats. It is
intermittent because it requires overlap between a busy Huabaosi run, user follow-up
timing, and WeCom send/format failure behavior.

## Resolution

No production mutation was made during diagnosis.

Recommended fix options:

- Add the busy-ack and formatting-fallback patterns to the live Hermes WeCom internal
  process filter through a reviewed release path.
- Or set Huabaosi `busy_input_mode` to `queue` or disable busy ack for WeCom after owner
  review.
- Keep any change in reviewed Hermes/release flow; do not hot-edit
  `/home/ubuntu/.hermes/hermes-agent` or the live profile directly.

## Validation

Completed:

- Confirmed `hermes-gateway-huabaosi.service` is active and owned by the Huabaosi Hermes
  profile.
- Located exact string generation in live Hermes source.
- Confirmed WeCom internal-process filter exists but lacks this pattern.
- Correlated the screenshot time window with WeCom send fallback failures and upstream
  model timeout.

Not completed:

- No patch was deployed to production.
- No full WeCom regression test was run against the live service.

## Remaining Boundary

Do not treat this as a Rust sidecar, QiWe image-send worker, or Huabaosi generated-image
artifact failure. It is a Hermes WeCom gateway busy-ack and send-format fallback issue.
